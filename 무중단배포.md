## 1. 전체 구조 개요

*본 시스템은 **추론 / 학습 / 배포를 완전히 분리**한 구조로 설계한다.*

- 추론: CPU 기반 EC2에서 상시 처리
- 학습: 요청 시에만 GPU 환경에서 실행 (SageMaker)
- 모델 저장: S3에 version 단위로 관리
- 배포: inference 서버에서 preload 후 active model 교체 (무중단)

---

## 2. 프론트 트리거 구조

### 1) 데이터 재수집

- DB → 학습 데이터 추출
- JSONL + manifest 생성
- S3 업로드
- dataset_version 생성

---

### 2) 재학습

- dataset_version 기반 학습 실행
- launcher job이 SageMaker Training Job 생성
- GPU 환경에서 학습 수행
- model artifact를 S3에 저장
- model_version 생성

---

### 3) 재배포

- 특정 model_version 선택
- inference 서버가 해당 모델 preload
- 검증 후 active model 전환

---

## 2.2 프론트 기준 실행 흐름

### 1) 데이터 재수집

1. 프론트 → POST /dataset-collections 호출
2. Backend → job 생성 (QUEUED)
3. dataset worker 실행 → RUNNING
4. S3 업로드 완료 → COMPLETED
5. 프론트는 GET /jobs/{job_id} 또는 SSE로 상태 확인

---

### 2) 재학습

1. 프론트 → POST /training-jobs 호출
2. Backend → job 생성 (QUEUED)
3. launcher 실행
4. SageMaker Training Job 생성
5. SageMaker 학습 수행
6. SageMaker → RabbitMQ 상태 이벤트 발행
7. Backend → MQ Consumer로 상태 수신 후 DB 업데이트
8. COMPLETED 시 model_version 저장

---

### 3) 재배포

1. 프론트 → POST /deployment-jobs 호출
2. Backend → job 생성 (QUEUED)
3. inference 서버 deploy 요청
4. preload → validate → switch
5. COMPLETED 시 ACTIVE 모델 반영

## 4. 학습 상태 전달 구조 (MQ 기반, 최종)

> SageMaker Training Job은 AWS VPC 내부에서 실행되며,
> 
> 
> VPN 터널을 통해 Backend 네트워크에 있는 RabbitMQ로 직접 상태 이벤트를 publish한다.
> 
> Backend는 RabbitMQ Consumer를 통해 이벤트를 수신하고 DB 상태를 업데이트한다.
> 

### 흐름

```
Backend → Launcher → SageMaker (Job 생성)
SageMaker → VPN → RabbitMQ → Backend (상태 이벤트)
```

---

### 이벤트 종류

```
RUNNING
COMPLETED
FAILED
```

### 로그 형식

```
user-id
sse-type
data 
```

x.sse.fanout 여기로 보냄 (dat에 실질적으로 관리자 화면 UI 로그에서 보고싶은 내용 담으면 됨)

---

### 역할 분리

- Launcher: Job 생성만 수행
- SageMaker: 상태 이벤트 발행
- Backend: MQ Consumer로 상태 반영

---

## 4.1 네트워크 구성

본 시스템은 AWS VPC와 Backend 서버 네트워크가 분리되어 있으며,
두 영역은 VPN 터널을 통해 연결된다.

### 네트워크 영역

Backend Network

- Java Backend
- RabbitMQ
- Launcher

AWS VPC

- SageMaker Training Job
- S3 (dataset / models)
- ECR
- Inference Server

### 연결 방식

AWS VPC ↔ VPN Tunnel ↔ Backend Network

## 14. 설계 핵심

- DB = 상태 판단
- SSE = 진행 표시
- MQ = 상태 이벤트 전달

## 15. 한 줄 정리

- 배포: Preload + Switch
- 상태: DB 최소화 + SSE
- 서비스: 무중단 유지
